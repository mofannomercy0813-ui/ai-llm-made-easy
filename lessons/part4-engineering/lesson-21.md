# 第 21 课：推理优化全家桶

大模型训完了。你兴高采烈地打开网页，输了一句话。

等了五秒钟。它在转圈。

GPT-4 每生成一个 Token 可能要花十几毫秒。你的「请帮我写一份周报」生成了 500 个 Token。你等了快十秒。这就是大模型推理的宿命，训得快不代表跑得快。

推理比训练更痛苦的原因在于，训练可以并行，一个 batch 里几百个样本一起算。推理必须串行，每次只生成一个 Token，下一个 Token 依赖上一个。GPU 空着等。

第一招。KV Cache。

每个 Token 在生成时都要做 Attention，而 Attention 需要 Q 和 K 计算相似度。在自回归生成中，已有 Token 的 K 和 V 每次都要重新算一遍，明明上一次迭代已经算过了，这次还得再来一遍。浪费。

KV Cache 就是把已经算过的 K 和 V 缓存起来。下一个 Token 生成时，只需为这个新 Token 计算 Q、K、V，把新 K 和 V 追加到缓存后面。已有的 Token 直接用缓存，不重新算。

显存占用大幅降低。但 KV Cache 本身占显存，每一层的每一个头都要存。序列越长，Cache 越大。对于 128K Token 的长序列，KV Cache 可能比模型本身还大。

第二招。vLLM 和 PagedAttention。

传统推理框架中，KV Cache 被预分配了一大块连续显存，不管实际用了多少。浪费严重。PagedAttention 把 KV Cache 像操作系统的内存分页一样切成小块。需要时分配一页，用完就回收。显存碎片化大幅减少，吞吐提升好几倍。vLLM 是目前最主流的推理引擎之一。

第三招。量化。第五课聊过了。推理时把模型从 FP16 量到 INT8 或 INT4，显存砍半甚至砍到四分之一，推理速度翻倍。代价是可接受的精度损失。

第四招。长上下文。之前说过原版 Transformer 的注意力复杂度 O(n²)。128K Token 的序列就是 160 亿次运算，一次推理就要这么多。FlashAttention 已经是标配了，不完整写出 n×n 矩阵，分块算，大幅减少 HBM 读写。另外还有稀疏注意力，不是每个 Token 都要看所有其他 Token。局部窗口加少量全局 Token，也能把 n² 压到 n×k。

第五招。投机解码。用一个小模型快速生成几个候选 Token，然后用大模型一次性验证这几个候选是否正确。对的多就全收，错的就扔掉重来。因为大模型做验证（并行的矩阵乘法）比生成（串行的逐个 Token）快得多。在低熵任务上，比如写代码、翻译，投机解码有时候可以提速两到三倍。

推理优化没有银弹。每个方案都是权衡。KV Cache 省计算但吃显存。量化省显存但掉精度。投机解码快但不稳定。你需要根据场景选组合。

## 追问模块

**追问 1：「KV Cache 有多大？」**

每层的每头缓存。大小 = 2 × 层数 × 头数 × d_head × 序列长度 × 精度字节数。GPT-3 175B，层数 96，头数 96，d_head=128，seq_len=2048，FP16 精度。粗略算约 2 × 96 × 96 × 128 × 2048 × 2 ≈ 9.6 GB。快赶上一个 7B 模型的参数量了。

**追问 2：「PagedAttention 的页多大？」**

通常跟操作系统的页大小对齐，4KB 或 16KB。太小管理开销大。太大碎片化解决得不够好。4KB 默认值在大多数场景下效果不错。

**追问 3：「量化对推理速度的影响到底多大？」**

INT8 相比 FP16，显存约减半，吞吐约增 1.5 到 2 倍。INT4 相比 FP16，显存减到四分之一，吞吐增 2 到 3 倍。但 INT4 需要更贵的反量化操作，实际加速比略低于理论值。
