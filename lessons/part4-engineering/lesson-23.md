# 第 23 课：MCP / A2A / Function Call，让模型会叫外援

上节课我们聊了 Agent 怎么决策和执行。但执行需要一个东西，工具。模型要去搜索引擎里查资料，它得有一个「叫搜索引擎」的接口。

三种方式。Function Call。MCP。A2A。

Function Call 是最基础的。模型在生成文本时，输出一段 JSON 描述它想调什么函数、传什么参数。外部的 Agent 框架截获这段 JSON，执行真正的函数，把结果喂回模型。像一个遥控器，模型按按钮，外面的人去操作。

优点极简。缺点是一次性的。模型每次调用工具都需要重新描述它要什么，没有持久化的工具记忆和管理机制。

MCP 就是来填这个坑的。Model Context Protocol，模型上下文协议。Anthropic 2024 年发布的开放标准。它的目标是让任何 AI 应用能用统一的方式发现和使用外部工具。

MCP 的核心架构是客户端-服务器。AI 宿主（比如 Claude Desktop）作为 MCP 客户端。每个工具或数据源封装成一个 MCP 服务器。服务器暴露标准化的接口，列出可用资源、描述工具的参数格式、处理请求。客户端通过 MCP 协议跟服务器通信。

MCP 的杀手锏在于标准化。你写一个 MCP 服务器连接你的公司数据库，Claude 能用，任何支持 MCP 的 AI 应用都能用。工具一次开发，到处复用。社区里已经有数百个现成的 MCP 服务器，Google Drive、GitHub、Slack、PostgreSQL。拿来即用，无需配置。

A2A 是 Agent-to-Agent 协议。Google 2024 年发布的。MCP 解决的是模型与工具之间的通信。A2A 解决的是 Agent 与 Agent 之间的通信。

不同团队、不同公司开发的 Agent 需要互相对话。你的采购 Agent 需要跟供应商的库存 Agent 确认交货时间。A2A 定义了 Agent 之间发现、握手、任务分配、结果回传的标准流程。跨组织的互操作性是它的核心价值。

Skill 是一个更轻量的概念。说到底就是一段预定义的 Prompt 加调用规范，告诉模型「当你看到用户想做什么时，按这套流程走，调这个工具」。MCP 是通用工具层，Skill 是场景化的工具编排。两者互补。

至于 MCP Server 怎么搭，通常用 Python 的 FastMCP 或 TypeScript 的 SDK。几步就能搭一个简单的 Server。但生产级的 Server 涉及认证鉴权、错误处理、超时重试、状态管理等一堆工程问题。

## 追问模块

**追问 1：「MCP 和传统 API 有什么不同？」**

传统 API 是工具自己定义的接口格式，每个工具一种格式，集成 N 个工具要看 N 份文档。MCP 是统一的协议层，所有工具用同一种方式暴露自己，客户端用同一种方式发现和调用。一次适配，覆盖所有。

**追问 2：「A2A 跟 MCP 冲突吗？」**

不冲突。MCP 是模型和工具之间的桥梁。A2A 是 Agent 和 Agent 之间的桥梁。Google 和 Anthropic 在设计时有意让它们互不重合。一个系统里可以同时用 MCP 做工具调用、用 A2A 做 Agent 协同。

**追问 3：「Function Calling 和 MCP 能不能互换？」**

Function Calling 是 MCP 的底层传输机制之一。MCP 比 Function Calling 多了一层工具发现和标准化描述。Function Calling 是模型输出的一句「我想调某某函数」。MCP 多了一步「先看看周围有哪些工具可用，再决定调哪个」。

## 思考题

1. 你手上有三个工具，邮件系统、日历、Jira。用 MCP 把它们封装起来。你会怎么设计每个 Server 的接口。

2. 两个公司的 Agent 要通过 A2A 协同，采购 Agent 和供应商 Agent。你设计一个通信协议，至少要包含哪些字段和流程。
