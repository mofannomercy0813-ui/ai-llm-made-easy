# 第 3 课：你怎么跟模型说话，Prompt + CoT + ICL

我朋友上个月第一次用 ChatGPT。他输入了四个字。

「写个方案。」

ChatGPT 给他生成了两页纸，关于一个完全不知道是什么的方案。他看了一眼，说这玩意不行。

我跟他说，你这不是在用 AI，你这是在跟一个刚入职、没见过你的、不知道你公司做什么的实习生说话。你跟他说「写个方案」，他能写出来才有鬼。

你跟大模型说话的方式，决定了它给你什么。

这东西有个正经名字，叫 Prompt Engineering。听起来很唬人，其实就是学会怎么给模型下指令。

---

说真的，Prompt Engineering 不是什么高深的技术。它更像一种手艺。

你想想看，你点外卖的时候也不会只写「吃的」两个字。你会写「一份大份黄焖鸡，微辣，多加土豆，不要香菜」。Prompt 就是这个意思，只不过你点的不是外卖，你点的是内容。

有个框架特别好记，叫 CRISP。

C 是 Context，上下文。你得告诉模型它现在是谁、在什么场景下、面对什么人。「你是一个有十年经验的 Python 后端工程师，正在给实习生写代码 review 意见。」

R 是 Role，角色。让模型扮演一个特定身份，比直接下指令效果好得多。「你是一个毒舌但真诚的美食评论家」比「写一个关于这家餐厅的评价」强一百倍。

I 是 Instruction，指令。你要它做什么，做到什么程度，什么格式。越具体越好。「写 300 字，用 Markdown 格式，分三段，每段一个小标题。」

S 是 Style，风格。语气、文风、复杂度。「用初中生能看懂的语言」「模仿鲁迅的杂文风格」「写得像小红书种草文案」。

P 是 Purpose，目的。你要这个回答拿去干什么。这会影响它选什么信息、用什么语气。「这份材料要发给投资人看的」「这是一个内部团队分享的 PPT」。

你把这五个要素配齐了，Prompt 的质量会直接上一个台阶。我自己的习惯是，每次写 Prompt 之前先在脑子里过一遍这五个字母，过完了再写。习惯之后，这就是本能反应了。

---

但有时候你就算 Prompt 写得再好，模型还是算不对。

比如「356 乘以 789 等于多少？」

模型可能会给你一个非常自信但完全错误的答案。不是它不努力，是因为它不擅长一口气做多步推理，它是在「猜」这个计算结果长什么样，而不是真的在算。

这时候你就需要 Chain-of-Thought，思维链。简称 CoT。

做法极其简单，就一句话，加在你的 Prompt 末尾。

「让我们一步步思考。」

就这么五个字。

然后模型就会像考试时在草稿纸上写演算过程一样，一步步拆解。「首先，356 乘以 700 等于 249200。然后，356 乘以 80 等于 28480。最后，356 乘以 9 等于 3204。三个加起来，249200+28480+3204=280884。」

准确率直接从蒙变成了对。

为啥呢。

因为当你让模型「一步步思考」的时候，它在生成每一步的时候都在贪婪地预测下一步应该是什么。第一步算对了，第二步建立在第一步之上，错的概率就小很多。如果让它一口气输出答案，它可能在第二步就算错了，然后后面的全完蛋。

CoT 不仅限于数学。任何需要推理的任务，逻辑题、代码调试、多条件分析，加上「让我们一步步思考」，效果都显著提升。

有意思的是，这句话不是研究者刻意设计出来的人造指令。是他们发现模型自己就会这么做，你只要提醒它而已。

还有一种更有意思的玩法。

你不需要告诉模型「怎么思考」，你直接给它看几个例子。

比如你想让它帮你把「我今天好累啊」翻译成英文。你不写 Prompt，你就直接在对话里贴两行。

「今天天气真好 → The weather is really nice today。这本书太无聊了 → This book is so boring。我今天好累啊 →」

模型会自动接上「I'm so tired today」。

这就是 In-Context Learning，上下文学习。你不用告诉它规则，你给它看几个例子，它自己就能学会模式。你不需要微调，不需要改参数，不需要训练任何东西。

输入几个例子。模型自动泛化。

ICL 的关键在于例子的质量和多样性。如果你的例子太单一，模型学会的模式就很窄。如果你的例子质量太差（比如翻译水平很烂），模型学出来的也好不到哪去。

而且 ICL 跟你放入例子的顺序也有关系。把相似类型的例子放一起，比把不同类型的例子混着放效果更好。这是个经验规则，还没有完美的理论解释，但它确实有效。

你想想看，这其实挺神奇的。一个有 1750 亿参数的模型，你不需要动它任何一个参数，只要在输入里给它看两三个例子，它就能立刻学会你想要的模式。这就像一个从来没学过法语的人，你给他看了三个英语翻译成法语的例子，他突然就能开始翻译了。

涌现又来了一次。

---

说真的，我觉得 Prompt Engineering、CoT、ICL 这三样东西，放在一起，是普通人能发挥大模型最大威力的组合拳。

先给上下文和角色，让模型进入状态。然后给一两个好的例子，让它知道你要什么格式。最后在指令里加上一句「请一步步推理」，让它不要省略中间过程。

你会发现在这套组合拳下，模型的输出质量会有一个肉眼可见的飞跃。

屏幕前的你，哪怕你现在什么 Prompt 技巧都不会，只记住两件事就够了。

第一，把要求写具体。不要「写个方案」，要「写一个针对大学生的校园奶茶店的营销方案，预算 5000 元，分线上和线下两个渠道，各三项具体措施」。

第二，需要推理时加上「让我们一步步思考」。

就这两条，能解决你 80% 的使用场景。

剩下的 20%，靠 CRISP 框架、例子引导和 RAG（第 20 课）来补充。

好了。你现在会用模型了。

## 追问模块：面试官可能怎么问

**追问 1：「Prompt Engineering 是工程学还是玄学？」**

目前更接近工程学，有系统的方法论（CRISP 框架、少样本学习、迭代优化），也有大量的量化实验支撑。但它依然依赖经验直觉，同样的 Prompt 在不同的模型上可能效果完全不同。正在变得越来越工程化，但还没到「保证可复现」的程度。

**追问 2：「CoT 和直接要求输出详细过程的区别在哪？」**

CoT 的关键在于「推理步骤的顺序」。如果你要求模型「输出详细过程」但不指定方向，它可能先猜答案再编过程。CoT 的「一步步来」强制模型按顺序推理，先算 A，基于 A 算 B，基于 B 得出结论。逻辑链条一旦断裂，错误就会暴露。

**追问 3：「ICL 本质上是不是微调的一种近似？」**

有个理论叫「meta-learning during inference」。模型在预训练时见过海量的模式，ICL 只是在推理时通过示例激活了跟示例模式最接近的「已存在」能力。但最近也有一些研究认为 ICL 确实在做隐式的梯度更新，这是一个还在争论的问题。面试时说「当前没有共识」就够了。

**追问 4：「Prompt Engineering 工程师这个岗位的护城河是什么？」**

坦率的讲，入门门槛极低。但做好的门槛极高。好的 Prompt 工程师靠的是对模型行为模式的深度直觉、对下游任务的精准理解、以及系统化的调试能力。一个月薪 5000 的 Prompt 工程师和一个月薪 5 万的差距，不在于谁写的 Prompt 更长。

## 思考题

1. 打开一个 AI 对话工具，用同一个任务（比如「帮我写一封请假邮件」）分别试两种 Prompt：一种只写「帮我写请假邮件」，一种用 CRISP 框架写完整版。对比两个结果，差距最大的是什么维度？

2. CoT 对数学和逻辑有用，但对写诗或者讲故事有用吗？为什么？试着找一个你觉得 CoT 完全帮不上忙的任务，分析一下原因。
